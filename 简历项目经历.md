# 短链通 - 短链接平台（简历项目经历）

## 项目经历

**短链通 - 短链接平台**
后端开发 | 2025-10 至 2025-12

**技术栈**：Go (gin/go-redis) + Java (Spring Boot/Spring Cloud Alibaba) + MyBatis + Redis + RocketMQ + ShardingSphere + ClickHouse + Nacos + Sentinel + Docker + Vue 3

**项目简介**：
短链通是面向企业和个人用户的短链接管理与分析平台。系统采用 Go 网关 + Java 微服务混合架构，Go 负责高性能流量入口、反向代理、全局限流和链路追踪，Java 负责短链接核心业务逻辑。平台提供短链接创建、管理、跳转统计、数据分析和全链路日志查询能力。

**项目亮点**：

1. **Go 网关替换 Spring Cloud Gateway**：使用 gin + 自定义 reverse proxy 实现高并发网关入口，配合 Nacos 服务发现实现热更新。网关层 TraceID 自动生成与透传，覆盖网关→Java 微服务→日志全链路。

2. **全局限流体系**：网关层实现内存令牌桶（O(1) 时间复杂度）+ Redis 分布式限流兜底，支持按路径前缀配置 QPS 阈值。短链接创建接口 QPS 可达 2600，较分布式锁方案提升 520 倍。

3. **ClickHouse 日志系统**：独立 Go 日志收集服务，基于 ClickHouse 实现百万级日志秒级查询。支持按时间范围、服务名、级别、关键词、TraceID 多维度过滤，配套 Vue 3 + Element Plus 日志查询前端和 Trace 链路追踪视图。

4. **高并发指标监控**：网关内嵌 Metrics Collector，基于 atomic + histogram 桶实现零锁埋点，支持 QPS、P50/P90/P99 延迟、错误率实时查询，5 秒自动刷新大屏。

5. **Bloom Filter 短路判断**：判断短链接是否存在，避免分布式锁竞争，性能远超传统方案。短链接创建接口 QPS 可达 2600，而传统分布式锁方案 QPS 仅为 5，性能提升显著。

6. **RocketMQ 削峰填谷**：借助消息队列的"削峰"特点实现海量访问短链接场景下的监控信息异步存储，确保系统在高负载情况下仍能正常运行。

7. **分片互斥策略**：优化更新或失效场景下大量请求查询数据库问题。封装缓存不存在时的读取功能，采用分布式锁互斥策略，减少对数据库的频繁访问。经 500 个短链接的 200 次随机访问，缓存命中率达到 95%+。

8. **缓存一致性**：通过更新数据库删除缓存的策略，保证短链接缓存与数据库之间的数据一致性。

9. **Redis 幂等消费**：在消息队列消费业务中，使用 Redis 完成幂等处理，确保消息在一定时间内仅被消费一次，避免重复处理。

10. **Redisson 分布式读写锁**：实现海量访问场景下的数据修改功能，确保数据修改的安全性和一致性。

11. **路由表分页**：在短链接数据分片基础上增加路由表，使用户能够方便地分页查看短链接。

12. **Sentinel 流量防护**：对接口访问 QPS 限流，保障系统稳定运行。当触发限流规则时，系统能进行降级处理，确保核心功能可用性。

## 面试高频问题 & 回答

### Q1: 为什么用 Go 重写网关？

> Spring Cloud Gateway 基于 Spring WebFlux，启动内存 700M+，单实例 QPS 约 3000。Go 网关内存仅 50M 左右，单实例 QPS 可达 3 万+，成本降了一个数量级。

### Q2: TraceID 全链路怎么实现的？

> Go 网关生成 TraceID 注入 `X-Trace-ID` header，反向代理自动传递给后端。Java 侧通过 Spring MVC Interceptor 读取 header 并注入 MDC，Logback 的 JSON 编码器自动输出。日志收集时带上 TraceID 写入 ClickHouse，日志查询时输入 TraceID 就能看完整调用链。

### Q3: 指标监控为什么用 histogram 不用存全部延迟？

> 如果每次请求都把延迟 append 到 slice，高并发下 slice 会无限增长，内存爆炸而且每次查询要排序。改成固定 11 个 histogram 桶（1ms-5s），每次请求只做 atomic 计数，零锁、O(1)、内存恒定。查 P99 时从桶累计推算，误差在桶粒度内，完全可接受。

### Q4: ClickHouse 写入瓶颈怎么解的？

> 原来用 `sync.Mutex` 锁 buffer，多 goroutine 写入串行化。改成 10000 容量的 channel 缓冲，写入只是 `ch <- entry`，零锁。后台单 goroutine 从 channel 消费，攒够 1000 条或 2 秒超时再批量写入 ClickHouse。配合 ClickHouse 的 `async_insert` 模式，写入性能大幅提升。

### Q5: 你们这个项目亮点是什么？

> 最大的亮点是 **Java 业务 + Go 网关的混合架构**。Java 做复杂业务（短链接管理、分库分表、分布式锁），Go 做高并发入口（网关、限流、日志收集）。既享受了 Java 生态的成熟，又拿到了 Go 的高性能。同时从网关到日志到前端都是我们自己从零实现的，不是调几个现成框架就能搞定的。

## 投递策略

### 三个方案对比

#### 方案 A：混合架构（Java + Go）— 推荐

**适合**：中级/高级后端、架构方向、或者岗位同时要求多语言

**优势**：差异化最大，能体现"选对工具的能力"，面试话题丰富
**劣势**：Go 部分如果没有上线数据支撑，面试官可能觉得只是练手

#### 方案 B：只写 Java

**适合**：投递纯 Java 岗位（阿里/字节/美团的 Java 岗）

就是原来的版本。稳妥，但差异化弱——Spring Cloud Alibaba + Redis + RocketMQ 是八股文标配，面试官见多了。

#### 方案 C：只写 Go

**适合**：投递 Go 岗位（很多中小厂、云原生公司）

但 Go 部分还没上线跑过真实流量，没有 QPS 数据背书，说服力偏弱。

---

### 建议

**写方案 A，但面试策略要有侧重**：

- **面 Java 岗** → 80% 聊 Java 亮点（分片、Bloom Filter、RocketMQ），Go 作为加分项："后来觉得网关太重，用 Go 重写了入口，内存从 700M 降到 50M"
- **面 Go 岗** → 80% 聊 Go 亮点（gin 网关、限流、ClickHouse、零锁 metrics），Java 部分简略带过
- **面架构/全栈** → 两边均衡讲，突出混合架构的选型思考
