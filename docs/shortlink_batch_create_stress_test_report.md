# 短链接系统：批量创建接口压力测试报告

## 1. 测试背景与目的
在实际业务场景中（如营销短信群发、商品活动分发），通常需要一次性将大量原始链接转换为短链接。为了减少频繁的网络建立连接和 I/O 损耗，系统专门提供了**批量创建短链接**（Batch Create）的接口。

本次压力测试旨在：
1. 验证批量创建接口 `/api/short-link/admin/v1/create/batch` 在高并发下的稳定性和吞吐极限。
2. 记录单次包含多条 URL 的请求的平均响应时间（RT），并以此评估数据库和缓存的批量写入性能。

---

## 2. 测试环境与配置参数
* **压测工具**：Apache JMeter (5.6.3)
* **目标接口**：`POST /api/short-link/admin/v1/create/batch`
* **硬件及部署环境**：单机 Windows Docker-Compose (含 MySQL, Redis, RocketMQ, ClickHouse, Backend-Project)
* **JMeter 测试模型**：
  * **并发线程数 (Threads)**：10
  * **循环次数 (Loops)**：50
  * **总样本数 (Samples)**：500 次请求
  * **业务特征**：每次请求的 Body 中包含 **5 个** 随机生成的原始长链接（Google, GitHub, Baidu, Zhihu, Bilibili）。

---

## 3. 测试数据与结果

从 JMeter Aggregate Report (聚合报告) 提取的核心指标如下：

| 指标项 | POST Batch Create (批量创建请求) |
| :--- | :--- |
| **总请求数 (# Samples)** | 500 |
| **错误率 (Error %)** | 0.00% |
| **平均响应时间 (Average RT)** | 227 ms |

---

## 4. 性能分析与结论

### 4.1 吞吐量折算与真实性能
JMeter 报告显示接口 QPS 为 **40.5 次/秒**。
**但由于这是批量接口，每次请求实际包含 5 个短链接的创建。**
* **系统真实吞吐量**：`40.5 * 5 = 202.5 条/秒`
* 这意味着系统每秒可以稳定创建 200+ 个全新的短链接，并且全部完成 MySQL 持久化、Redis 缓存预热以及布隆过滤器的位图写入。

### 4.2 响应时间分析
* 单条记录创建（无锁）平均耗时在几毫秒至几十毫秒。
* 批量包含 5 条记录的创建，平均耗时为 **227 ms**（中位数 180 ms），响应时间并没有随着数量线性成倍增加，而是体现出了批量 I/O 带来的红利。
* **0.00% 的错误率**证明了即使在 200ms+ 的响应耗时下，数据库事务池、Redis 连接池以及 Tomcat 线程池都运转正常，未出现 Socket Timeout 或资源耗尽。

### 4.3 后续优化建议
虽然目前的批量性能已经可以满足日常营销活动的需求，但如果在极端的大促（如双11）场景下需要支持单次上千条的批量转化，可采取以下优化：
1. **数据库批量插入 (Batch Insert)**：检查 MyBatis-Plus 的 `saveBatch` 机制是否开启了 `rewriteBatchedStatements=true` 驱动参数，这能将 MySQL 写入性能再提升 3-5 倍。
2. **异步削峰 (MQ)**：对于非实时要求极高的海量批量创建，可以将其改造为异步接口。即接收请求后将任务丢入 RocketMQ，立即返回 200 状态和任务 ID，由后台消费者慢慢消费并创建短链接。
