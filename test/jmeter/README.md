# Linkhub-SaaS 性能测试与面试演示指南

本手册旨在指导如何使用 JMeter 对 Linkhub-SaaS 进行压测，并在面试中展示系统的核心可观测性指标。

## 1. 前置准备

1. **安装 JMeter**: 推荐使用 JMeter 5.5 或更高版本。
2. **启动服务**: 确保后端工程（8001端口）已正常启动。
3. **指标查询接口**:
   - 聚合指标: `http://localhost:8001/api/short-link/v1/metrics/summary`
   - Actuator 指标: `http://localhost:8001/actuator/prometheus`

---

## 2. 测试脚本说明

| 文件 | 用途 |
| :--- | :--- |
| `redirect-cache-test.jmx` | 短链重定向缓存命中率、布隆过滤器拦截、指标监控 |
| `create-qps-test.jmx` | 单条短链创建 QPS（无锁 vs 有锁对比）、Sentinel 限流验证 |
| `batch-create-test.jmx` | **[新增]** 批量创建短链接性能测试（自动登录获取 Token → 批量提交） |
| `stats-concurrency-test.jmx` | **[新增]** 高并发重定向 + 并发查询统计，验证异步统计写入与读写分离 |
| `page-query-test.jmx` | **[新增]** 分页查询性能对比（按创建时间 vs 按 PV 排序），验证复杂 SQL 优化成果 |
| `bulk-delete-test.jmx` | **[新增]** 并发批量删除（移入回收站），模拟前端 `Promise.allSettled` 场景 |

## 3. 面试演示场景

### 场景 A：缓存命中率与重定向性能（最常用）

**面试问题**: "你的系统在高并发下重定向性能如何？缓存命中率有多少？"

1. **打开脚本**: `test/jmeter/redirect-cache-test.jmx`
2. **设置变量**: 在 `User Defined Variables` 中填入一个已存在的 `shortUri`。
3. **执行压测**: 启动 `Redirect (Valid - Cache Test)` 线程组。
4. **演示要点**:
   - **响应时间**: 查看 Aggregate Report，P99 应在 5-20ms 左右（本地环境）。
   - **命中率**: 访问 `/api/short-link/v1/metrics/summary`，观察 `cacheHitRate` 随压测进行稳定在 95% 以上。
   - **技术细节**: 提到系统采用了 **Redis 多级缓存策略**，避免了频繁查询数据库。

### 场景 B：布隆过滤器拦截（防穿透）

**面试问题**: "如果有人恶意攻击，请求大量不存在的短链接，你的系统怎么处理？"

1. **执行压测**: 启动 `Redirect (Invalid - Bloom Filter Test)` 线程组。
2. **演示要点**:
   - **拦截统计**: 刷新指标接口，观察 `bloomFilterInterceptCount` 快速上升。
   - **原理说明**: 提到系统在查询缓存/数据库前先通过 **Redisson 布隆过滤器** 拦截，有效防止了缓存穿透。

### 场景 C：分布式锁 vs 普通创建（性能对比）

**面试问题**: "创建短链接为什么要用分布式锁？用了锁之后性能损耗大吗？"

1. **执行压测**: 先后运行 `create-qps-test.jmx` 中的两个线程组。
2. **演示要点**:
   - **普通创建 (No Lock)**: QPS 更高，但在极端并发下可能出现 key 碰撞（由数据库唯一索引兜底）。
   - **带锁创建 (With Lock)**: QPS 略低（锁竞争开销），但保证了创建过程的强原子性。
   - **指标展示**: 通过 `RLock` 保证高并发 Suffix 碰撞时的正确性。

### 场景 D：Sentinel 限流验证

**面试问题**: "你的系统如何做流控？怎么知道限流生效了？"

1. **配置**: 确认 `SentinelRuleConfig` 中 `create_short-link` 的 QPS 为 1（默认）。
2. **执行压测**: 启动 `Normal Create Request`。
3. **演示要点**:
   - **错误码**: 观察 JMeter 响应结果，出现大量 `B100000` 错误码及"当前访问人数过多"提示。
   - **限流计数**: 刷新指标接口，观察 `sentinelBlockCount` 持续增加。

### 场景 E：批量创建短链接性能测试（新增）

**面试问题**: "批量创建接口性能如何？你们是怎么修复批量创建 Bug 的？前端如何优化？"

1. **打开脚本**: `test/jmeter/batch-create-test.jmx`
2. **设置变量**: 在 `User Defined Variables` 中填入 `username` / `password` / `gid`。
3. **执行步骤**:
   - 先单独运行 `Login - Get Token` 线程组（执行一次，将 Token 存入 JMeter 变量）。
   - 再启动 `Batch Create Short Links` 线程组进行压测。
4. **演示要点**:
   - **Bug 修复**: 原来 `smallLinkPage.js` 中 `addLinks` 设置了 `responseType: 'arraybuffer'`，导致后端返回的 JSON 被当作二进制流，触发空 Excel 下载。修复方式是删除该配置，恢复标准 JSON 模式。
   - **描述自动填充**: 批量创建弹窗现在会在用户停止输入 800ms 后，对每行 URL 并发调用 `/title` 接口自动填充描述信息（`Promise.all` 并发 + 按行索引对齐）。
   - **吞吐量对比**: 单次批量请求包含 5 条链接，实际吞吐量等效 5 倍单条 QPS，更适合大批量导入场景。
   - **Sentinel 配置**: 批量接口使用匀速排队（leaky-bucket）模式，高并发下不直接拒绝而是平滑排队，与单条接口快速失败模式形成对比。

### 场景 F：登录延迟与前端 Loading 体验（新增）

**面试问题**: "登录接口有一定延迟，前端是如何提升用户体验的？"

1. **演示要点**:
   - **后端延迟**: 登录接口涉及数据库查询 + Token 签发，本地约 50-300ms。
   - **前端优化**: 点击「登录」后立即将 `loading.value = true`，按钮进入 Element Plus 转圈禁用状态，通过 `try/finally` 确保无论接口成功/失败/超时，`loading` 都会在最终被重置为 `false`，避免按钮卡死无法再次点击。
   - **效果**: 用户获得明确的「正在登录中」视觉反馈，不会因等待而重复点击，减少了重复请求的风险。

| 指标项目 | 预期数值（本地开发环境） | 面试亮点描述 |
| :--- | :--- | :--- |

### 场景 G：高并发访问 + 统计读写并发验证（新增）

**面试问题**: "短链统计数据是同步写入还是异步写入？高并发下统计延迟如何？"

1. **打开脚本**: `test/jmeter/stats-concurrency-test.jmx`
2. **设置变量**: 填入已存在短链的 `shortUri`、`fullShortUrl`、`gid`。
3. **执行步骤**: 先运行 `Login - Get Token`，再同时启动「高并发重定向」和「并发统计查询」两个线程组。
4. **演示要点**:
   - **读写分离**: 100 线程并发访问短链（触发 PV/UV 异步统计写入）的同时，20 线程并发查询统计接口——两者互不阻塞，查看 Aggregate Report 中两个线程组的响应时间均保持稳定。
   - **异步设计**: 统计写入通过异步队列处理，不影响重定向主链路的 P99，这正是「读多写少异步化」的核心设计亮点。
   - **分组统计**: 顺带展示 `GET /stats/group` 接口的聚合查询能力。

### 场景 H：分页查询性能对比（新增）

**面试问题**: "按访问量排序分页和按时间排序分页，性能差多少？你们是怎么优化的？"

1. **打开脚本**: `test/jmeter/page-query-test.jmx`
2. **设置变量**: 填入 `gid`，确保数据库中有足够的测试数据（建议 100 条以上）。
3. **执行步骤**: 先运行 `Login - Get Token`，再分别启动两个分页线程组，对比 Aggregate Report 中两组的 P99。
4. **演示要点**:
   - **orderTag=0（按创建时间）**: 命中主键/创建时间索引，查询极快，P99 通常 < 10ms。
   - **orderTag=1（按 PV 排序）**: 需要 JOIN `t_link_stats` 表做统计聚合再排序，复杂度更高，P99 显著更高——正好引出索引优化、查询缓存、预计算等话题。
   - **缓存监控**: 同步运行的「缓存命中率监控」线程组验证监控接口本身轻量，不成为瓶颈。

### 场景 I：批量删除并发压测（新增）

**面试问题**: "前端批量删除是怎么实现的？后端能承受多用户同时批量删除的并发压力吗？"

1. **打开脚本**: `test/jmeter/bulk-delete-test.jmx`
2. **设置变量**: 填入真实存在的 `fullShortUrl1` 和 `gid1`（可从分页接口获取）。
3. **执行压测**: 先运行登录，再启动并发删除线程组（20 线程模拟 20 个用户同时批量删除）。
4. **演示要点**:
   - **前端实现**: 批量删除通过 `Promise.allSettled` 并发调用 `recycle-bin/save`，每条链接独立请求，部分失败不中断其他条目。
   - **后端承压**: 20 线程并发 50 轮，查看成功率和 P99，验证接口的幂等性（重复移入已在回收站的链接应返回友好错误而非崩溃）。
   - **操作链路**: 结合主页面多选 → 批量删除 → 回收站恢复的完整链路展示，体现系统可操作性。
