# Linkhub-SaaS 性能测试与面试演示指南

本手册旨在指导如何使用 JMeter 对 Linkhub-SaaS 进行压测，并在面试中展示系统的核心可观测性指标。

## 1. 前置准备

1. **安装 JMeter**: 推荐使用 JMeter 5.5 或更高版本。
2. **启动服务**: 确保后端工程（8001端口）已正常启动。
3. **指标查询接口**: 
   - 聚合指标: `http://localhost:8001/api/short-link/v1/metrics/summary`
   - Actuator 指标: `http://localhost:8001/actuator/prometheus`

---

## 2. 面试演示场景

### 场景 A：缓存命中率与重定向性能 (最常用)
**面试问题**: "你的系统在高并发下重定向性能如何？缓存命中率有多少？"

1. **打开脚本**: `test/jmeter/redirect-cache-test.jmx`
2. **设置变量**: 在 `User Defined Variables` 中填入一个已存在的 `shortUri`。
3. **执行压测**: 启动 `Redirect (Valid - Cache Test)` 线程组。
4. **演示要点**:
   - **响应时间**: 查看 Aggregate Report，P99 应在 5-20ms 左右（本地环境）。
   - **命中率**: 访问 `/api/short-link/v1/metrics/summary`，观察 `cacheHitRate` 随压测进行稳定在 95% 以上。
   - **技术细节**: 提到系统采用了 **Redis 多级缓存策略**，避免了频繁查询数据库。

### 场景 B：布隆过滤器拦截 (防穿透)
**面试问题**: "如果有人恶意攻击，请求大量不存在的短链接，你的系统怎么处理？"

1. **执行压测**: 启动 `Redirect (Invalid - Bloom Filter Test)` 线程组。
2. **演示要点**:
   - **拦截统计**: 刷新指标接口，观察 `bloomFilterInterceptCount` 快速上升。
   - **原理说明**: 提到系统在查询缓存/数据库前先通过 **Redisson 布隆过滤器** 拦截，有效防止了缓存穿透。

### 场景 C：分布式锁 vs 普通创建 (性能对比)
**面试问题**: "创建短链接为什么要用分布式锁？用了锁之后性能损耗大吗？"

1. **执行压测**: 先后运行 `create-qps-test.jmx` 中的两个线程组。
2. **演示要点**:
   - **性能对比**: 
     - **普通创建 (No Lock)**: QPS 更高，但可能在极端并发下出现重复 key 碰撞（由数据库唯一索引兜底）。
     - **带锁创建 (With Lock)**: QPS 略低（由于锁竞争），但保证了创建过程的强原子性。
   - **指标展示**: 提到系统通过 `RLock` 保证了在高并发 Suffix 碰撞时的正确性。

### 场景 D：Sentinel 限流验证
**面试问题**: "你的系统如何做流控？怎么知道限流生效了？"

1. **配置**: 确认 `SentinelRuleConfig` 中 `create_short-link` 的 QPS 为 1（默认）。
2. **执行压测**: 启动 `Normal Create Request`。
3. **演示要点**:
   - **错误码**: 观察 JMeter 响应结果，会出现大量的 `B100000` 错误码及“当前访问人数过多”提示。
   - **限流计数**: 刷新指标接口，观察 `sentinelBlockCount` 增加。

## 3. 预期指标基准 (参考值)

| 指标项目 | 预期数值 (本地开发环境) | 面试亮点描述 |
| :--- | :--- | :--- |
| **重定向 P99** | < 30ms | 极速响应，用户无感知 |
| **热点缓存命中率** | > 95% | 有效减轻数据库压力 |
| **布隆过滤器拦截率** | ~ 100% | 完美拦截非法请求，防护数据库 |
| **创建短链 QPS** | 200+ (取决于硬件) | 支撑高并发短链生成场景 |

---

> **小贴士**: 面试时可以一边开着 JMeter 跑压测，一边在浏览器实时刷新 `/metrics/summary` 接口给面试官看，数据跳动的视觉效果非常有说服力！
