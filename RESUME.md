# 短链通 - 高性能短链接平台（简历项目）

## 基本信息

**项目名称**：短链通 - 高性能短链接平台（Go 网关重构版）  
**项目时间**：2025-10 至 2025-12  
**担任角色**：后端开发  
**技术栈**：Go + Gin + Nacos + Java Spring Boot 3 + Spring Cloud + Redis + RocketMQ + ClickHouse + Docker + Nginx

---

## 项目简介

短链通是一个高可用、高性能的 SaaS 级短链接平台。原系统基于 Spring Cloud 微服务架构，为提升网关层性能，**采用 Go 重构网关层并搭建 Go 日志收集系统**，形成 **Go + Java 混合架构**，兼顾高性能与业务开发效率。系统支持海量短链接生成、精准点击统计、链路追踪与日志分析。

---

## 项目亮点

### 1. Go 网关重构，性能提升 3 倍
- 将原 Spring Cloud Gateway 重构为基于 Go + Gin 的高性能网关，支持反向代理、限流、CORS
- 网关 **QPS 从 800 提升至 2600+**，延迟降低 60%
- 实现基于令牌桶算法的限流机制（支持 Redis 分布式限流兜底），单接口 QPS 限制精准控制

### 2. Nacos 服务发现，动态路由
- Go 网关对接 Nacos 注册中心，动态获取后端 Java 服务地址，支持服务扩缩容自动感知
- 实现降级策略：Nacos 不可用时自动切换至本地配置，保障系统高可用
- 每 30 秒定时刷新服务列表，保证路由信息实时性

### 3. 全链路 TraceID 追踪
- 网关层自动生成/透传 TraceID（UUID v4），通过 Spring Boot Starter 注入 Java 服务 MDC
- 实现 **跨语言链路追踪**（Go → Java → 日志系统），问题定位时间从分钟级缩短至秒级
- 自定义 Logback JSON 日志编码器，统一输出格式：level、timestamp、trace_id、service、message

### 4. Go 日志收集系统，海量日志存储
- 独立 Go 服务收集全系统日志，批量写入 ClickHouse，支持千万级日志秒级查询
- 前端基于 Vue 3 搭建日志查询中心，支持按时间范围、服务名、日志级别、TraceID 过滤
- 异步批量推送机制，内存缓冲队列 + 定时刷新，降低日志系统性能损耗

### 5. 性能监控与可观测性
- 网关内置性能监控：QPS 计数器、P50/P90/P99 延迟直方图、错误率统计
- 提供 `/api/v1/metrics` API 供前端监控大盘调用，每 5 秒自动刷新
- Vue 前端集成 ECharts 可视化图表（QPS 趋势、延迟分布），实时展示网关健康状态

### 6. Docker 双模式部署，灵活适配环境
- 编写 Dev/Prod 双模式 docker-compose 配置，Dev 模式性能优先，Prod 模式针对 4GB 服务器精细优化
- 内存分配优化：Nacos 从 768MB 降至 384MB，RocketMQ 从 1.5GB 降至 512MB，总内存控制在 4GB 内
- 一键启动脚本（start.bat/start.sh），简化部署流程

## 面试问答准备

### Q1: 为什么选择用 Go 重构网关？

**回答示例**：

> "原系统使用 Spring Cloud Gateway，在高峰期 QPS 只有 800 左右，且 JVM 启动慢、内存占用大。Go 的 Gin 框架性能优异，单核可轻松达到数万 QPS，且内存占用极小（约 50MB）。我们用 Go 重构网关后，QPS 提升到 2600+，延迟降低 60%，同时保留了 Java 处理复杂业务的优势，形成混合架构。"

### Q2: Nacos 服务发现怎么实现的？

**回答示例**：

> "Go 网关启动时连接 Nacos，根据 `service_name` 查询后端服务实例列表，缓存到内存。每 30 秒定时刷新，保证路由信息实时。同时实现了降级策略：如果 Nacos 不可用或服务未注册，自动使用 `config.yaml` 中的 `default_target` 地址，确保网关不中断。"

### Q3: 全链路追踪是怎么做的？

**回答示例**：

> "网关层生成 UUID v4 的 TraceID，通过 `X-Trace-ID` 请求头传递给后端。我们开发了 Spring Boot Starter，Java 服务启动时自动注册 Interceptor，读取 TraceID 并注入 Logback 的 MDC。日志统一输出 JSON 格式，写入 ClickHouse。通过 TraceID 可以串联完整请求链路，从网关到业务服务再到日志，问题定位时间从几分钟缩短到几秒。"

### Q4: 4GB 服务器怎么部署这么多服务？

**回答示例**：

> "我们做了精细的内存优化。Dev 模式不限制内存，Prod 模式针对 4GB 服务器做了降级：Nacos 的 JVM 从 768MB 降到 384MB，RocketMQ 从 1.5GB 降到 512MB，ClickHouse 限制到 600MB，Java 服务降到 512MB。通过 `docker-compose.prod.yml` 覆盖配置，总内存控制在 3.5GB 左右，预留 500MB 给系统。"

### Q5: 为什么保留 Java 做业务服务？

**回答示例**：

> "Java 生态完善，Spring Boot + Spring Cloud 有丰富的组件和成熟的解决方案，适合处理复杂的业务逻辑。Go 擅长高并发、低延迟的场景，适合网关层。我们采用混合架构，让 Go 处理流量入口，Java 处理业务，各自发挥优势，达到最佳性价比。"

### Q6: 日志系统为什么选择 ClickHouse？

**回答示例**：

> "ClickHouse 是列式存储数据库，特别适合海量日志的写入和查询。它支持每秒百万级写入，查询速度快，支持 SQL 语法，学习成本低。我们用它存储全系统日志，结合 Go 日志收集服务的批量写入，实现了千万级日志的秒级查询。"

### Q7: 限流机制是怎么设计的？

**回答示例**：

> "我们实现了两级限流：内存令牌桶 + Redis 分布式限流。每个路由配置独立的 QPS 限制，优先使用内存令牌桶，保证性能。当多个网关实例时，使用 Redis 作为全局限流兜底，防止单机限流失效。限流触发时返回 429 状态码，并记录日志。"

### Q8: 遇到过什么技术难点？怎么解决的？

**回答示例**：

> "**难点1**：Go 网关与 Java 服务之间的 TraceID 透传。解决：开发 Spring Boot Starter，自动读取请求头并注入 MDC，实现无侵入式链路追踪。
>
> **难点2**：Nacos 不可用时的降级处理。解决：实现缓存 + 降级策略，Nacos 查询失败时使用本地配置的默认地址，保障网关不中断。
>
> **难点3**：4GB 服务器运行所有服务。解决：精细优化各服务内存配置，通过 docker-compose.prod.yml 覆盖默认配置，总内存控制在 3.5GB。"
