## 二、核心技术亮点

### 2.1 缓存穿透防护（布隆过滤器）

**可能的问题：**
1. **布隆过滤器的原理是什么？**
   - 位数组 + 多个哈希函数
   - 存在：可能误判；不存在：一定准确
   - 本项目：1 亿容量，0.001 误判率

2. **为什么选择 0.001 的误判率？如何计算出来的？**
   - 误判率公式：`p = (1 - e^(-kn/m))^k`
   - 1 亿数据量，0.001 误判率需要约 19 亿位（约 238MB）
   - 哈希函数次数 k ≈ 14 次

3. **布隆过滤器数据如何初始化？**
   - 项目启动时从数据库加载已存在的短链接
   - 使用 Redisson 的 `tryInit()` 方法初始化
   - 新增短链接时动态添加到布隆过滤器

4. **如果布隆过滤器宕机重启怎么办？**
   - 持久化问题：Redisson 布隆过滤器存储在 Redis，重启不丢失
   - 缓存穿透风险：重启期间可能短暂失效

5. **为什么不用本地缓存（如 Caffeine）？**
   - 分布式环境需要共享缓存
   - 微服务多实例部署

**代码位置：**
```java
// RBloomFilterConfiguration.java
public RBloomFilter<String> shortUriCreateCachePenetrationBloomFilter(RedissonClient redissonClient) {
    RBloomFilter<String> cachePenetrationBloomFilter = redissonClient.getBloomFilter("shortUriCreateCachePenetrationBloomFilter");
    cachePenetrationBloomFilter.tryInit(100000000L, 0.001);
    return cachePenetrationBloomFilter;
}
```

### 2.2 高并发限流（Redis + Lua + Sentinel）

**可能的问题：**
1. **你们的限流策略是如何实现的？**
   - 两层限流：
     - 用户操作限流：Redis + Lua（1 秒 20 次请求）
     - 接口访问限流：Sentinel（100QPS）

2. **为什么用 Lua 脚本？优势是什么？**
   - 原子性操作：INCR + EXPIRE 保证原子性
   - 避免并发问题：防止时间窗口边界条件
   - 减少网络开销：一次请求完成

3. **滑动窗口算法的原理？**
   ```lua
   -- 核心逻辑
   local currentAccessCount = redis.call("INCR", accessKey)
   if currentAccessCount == 1 then
       redis.call("EXPIRE", accessKey, timeWindow)
   end
   return currentAccessCount
   ```

4. **Sentinel 的限流规则如何配置？**
   - 基于 QPS 模式
   - 自定义 BlockHandler 返回友好提示
   - 支持热点参数限流

5. **限流后如何降级？**
   - 返回友好提示："当前访问人数过多，请稍后再试"
   - 记录日志，便于后续优化

**代码位置：**
```java

Long result = stringRedisTemplate.execute(redisScript,
```

### 2.3 缓存与数据库一致性

**可能的问题：**
1. **如何保证缓存和数据库的数据一致性？**
   - 策略：**更新数据库 + 删除缓存**
   - 原因：避免并发写导致的不一致

2. **为什么不用"先更新数据库，再更新缓存"？**
   - 并发场景下可能产生脏数据
   - 示例：线程 A 更新 DB → 线程 B 更新 DB → 线程 B 更新 Cache → 线程 A 更新 Cache

3. **删除缓存失败怎么办？**
   - 分布式锁：Redisson RLock 保证原子性
   - 重试机制：多次尝试删除
   - 最终一致性：通过 binlog 异步同步（Canal）

4. **不一致率 0.1% 是如何测试出来的？**
   - 使用 JMeter 进行并发测试
   - 模拟 1000 并发，持续 1 分钟
   - 对比缓存与数据库数据差异

5. **缓存穿透、击穿、雪崩的区别？**
   - 穿透：查询不存在的数据 → 布隆过滤器
   - 击穿：热点 key 过期 → 分布式锁
   - 雪崩：大量 key 同时过期 → 随机 TTL

**代码位置：**
```java
// ShortLinkServiceImpl.java
RLock lock = redissonClient.getLock(LOCK_GOTO_SHORT_LINK_KEY);
if (lock.tryLock()) {
    try {
        // 1. 更新数据库
        shortLinkMapper.incrementStats(...);
        // 2. 删除缓存
        stringRedisTemplate.delete(GOTO_SHORT_LINK_KEY);
    } finally {
        lock.unlock();
    }
}
```

### 2.4 异步数据统计（Redis Stream）

**可能的问题：**
1. **为什么选择 Redis Stream 而不是消息队列（如 RocketMQ）？**
   - 轻量级：无需额外部署
   - 低延迟：微秒级消费
   - 持久化：支持消息持久化和消费者组

2. **Redis Stream 的消息队列模型？**
   - Producer：访问日志写入 Stream
   - Consumer：消费者组并发消费
   - ACK 机制：保证消息不丢失

3. **如何保证消息不重复消费？**
   - 幂等性处理：使用唯一消息 ID
   - Redis 记录已处理消息：2 分钟过期
   - 消费者组自动去重

4. **如何保证消息不丢失？**
   - 持久化：AOF + RDB
   - ACK 机制：消费成功后确认
   - 异常重试：删除幂等标识，重新入队

5. **PV/UV/UIP 是如何统计的？**
   - PV：每次访问 +1
   - UV：基于用户标识（去重）
   - UIP：基于 IP 地址（去重）
   - 使用 HyperLogLog 或 Set 去重

**代码位置：**

## 三、数据库与分库分表

### 3.1 ShardingSphere 分片策略

**可能的问题：**
1. **分库分表是如何设计的？**
   - 分库：16 个物理库（t_user_0 ~ t_user_15）
   - 分片键：username / gid / full_short_url
   - 分片算法：HASH_MOD（哈希取模）

2. **为什么选择 16 个分片？**
   - 数据量预估：单表 1000 万数据
   - 扩展性考虑：2 的幂次方，便于扩容
   - 性能平衡：避免单表过大

3. **非分片键查询如何处理？**
   - 问题：按日期查询需要路由到所有分片
   - 解决：建立映射表或使用 ES

4. **分库分表后的 ID 生成策略？**
   - 雪花算法（Snowflake）
   - 保证全局唯一、趋势递增

5. **数据迁移和扩容方案？**
   - 双写方案：新旧库同时写入
   - 数据同步：增量同步历史数据
   - 切换读写的灰度发布

**配置文件：**
```yaml
# shardingsphere-config-dev.yaml
rules:
  - !SHARDING
    tables:
      t_user:
        actualDataNodes: ds_0.t_user_${0..15}
        tableStrategy:
          standard:
            shardingColumn: username
            shardingAlgorithmName: user_table_hash_mod
    shardingAlgorithms:
      user_table_hash_mod:
        type: HASH_MOD
        props:
          sharding-count: 16
```
