## 背景

<!-- 背景和当前状态 -->
本项目目前在 Java 微服务和 Go 网关中虽有部分日志记录，但缺乏统一的、覆盖全方法的详细调试日志。Java 端目前没有统一的 AOP 日志切面，Go 端网关虽有初步的访问日志（Access Logger），但功能较为简单，且缺乏对内部函数调用链的详细追踪。

## 目标 / 非目标

**目标:**
<!-- 本设计旨在实现的目标 -->
-   **全量覆盖**: 实现 Java 模块（Controller/Service 层）的自动化 AOP 日志记录。
-   **环境敏感**: 在开发环境下提供详尽的 `DEBUG` 级别日志，在生产环境保持高效的 `INFO` 级别。
-   **统一格式**: 规范化 Java 和 Go 服务的日志输出格式，包含 TraceID 以便全链路追踪。
-   **安全性**: 实现自动化的敏感字段（密码、令牌等）脱敏。
-   **可观测性**: 记录方法执行耗时，帮助定位性能瓶颈。

**非目标:**
<!-- 明确不在范围内的内容 -->
-   不涉及日志的长期存储架构设计（如 ELK 集群搭建）。
-   不涉及前端页面的日志埋点。

## 设计决策

<!-- 关键设计决策及理由 -->
### 1. Java 端实现：Spring AOP
-   **决策**: 使用 `shortlink-trace-starter` 模块定义一个全局切面类 `LogAspect`。
-   **理由**: 这种方式对业务代码零侵入，通过简单的配置即可覆盖整个项目的核心逻辑层。
-   **实现方案**:
    *   使用 `@Around` 环绕通知捕获方法执行。
    *   利用 `ProceedingJoinPoint` 获取入参、方法名和执行结果。
    *   通过自定义注解 `@NoLog`（可选）排除不需要记录日志的特定方法。

### 2. Go 端实现：Zap 升级与中间件增强
-   **决策**: 升级 `shortlink-gateway-go` 的日志系统，从标准库 `log` 全面转向 `uber-go/zap`。
-   **理由**: `zap` 提供结构化日志输出，性能极高，且易于集成敏感信息过滤。
-   **实现方案**:
    *   重构 `internal/middleware/traceid.go` 中的 `AccessLogger`，使用结构化日志记录。
    *   在 `config.yaml` 中增加日志级别和格式配置。

### 3. 脱敏策略
-   **决策**: 基于字段名匹配的拦截式脱敏。
-   **理由**: 简单有效，能覆盖 90% 以上的敏感数据泄露场景。
-   **实现方案**:
    *   Java: 在 AOP 处理入参和出参时，通过反射或 JSON 序列化过滤器（如 Jackson 的 `SimpleBeanPropertyFilter`）检查字段名并替换。
    *   Go: 在 Zap 的 Core 层添加自定义 `Encoder` 或在中间件处理 Map 时进行替换。

## 风险 / 权衡

<!-- 已知风险和权衡 -->
-   **性能损耗**: 频繁的反射和 JSON 序列化会带来一定的 CPU 开销。
    *   *对策*: 仅在 `DEBUG` 级别或开发环境下记录完整的对象 JSON，生产环境仅记录基础元数据。
-   **磁盘空间**: 详细的日志会迅速消耗磁盘空间。
    *   *对策*: 严格配置 `Lumberjack` (Go) 和 `Logback` (Java) 的文件滚动策略（如保留 7 天，单个文件最大 100MB）。
