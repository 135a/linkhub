## Context

当前代码仓库包含多个后端模块（如 `admin`、`project`、`gateway`），与目标“独立 Spring Boot 单体应用”不一致，导致构建链路、配置项和运行边界复杂。  
本次设计以“仅保留一个可独立运行的 `main` 模块”为核心约束，并明确以下硬性前提：

- 不引入或保留网关层（无 `gateway` 模块）。
- 不依赖 Nacos 或其他注册/配置中心。
- 不依赖任何远程服务调用（如 Feign、Dubbo、RPC 等）。
- 所有业务能力在单 JVM 进程内完成，模块间调用收敛为本地代码调用。

## Goals / Non-Goals

**Goals:**

- 将后端收敛为单模块：仅保留 `main` 作为唯一业务模块与唯一启动入口。
- 完成 `admin`、`project` 中仍需保留的业务逻辑迁移到 `main`，并清理重复实现。
- 删除无关模块及其依赖关系，确保 Maven 聚合结构清晰、可维护。
- 将配置、构建、部署、文档统一到单体模型：单服务启动、单服务测试、单服务交付。
- 在不破坏核心短链接能力的前提下完成架构收敛，保留必要的可测试性与可观测性基线。

**Non-Goals:**

- 不建设微服务治理能力（注册发现、服务熔断、服务路由等）。
- 不引入分布式配置中心、消息总线或跨服务事务方案。
- 不在本次变更中重构全部业务域模型，仅做与单体化直接相关的必要调整。
- 不新增多套运行模式（如单体/微服务双模并存）。

## Decisions

1. **单模块主干决策：仅保留 `main`**
   - 方案：将可复用业务代码迁移到 `main`，删除 `admin`、`project`、`gateway` 等冗余模块（其中 `gateway` 直接移除，不做替代模块）。
   - 理由：单一模块可以显著降低认知成本、构建复杂度和跨模块耦合风险。
   - 备选方案：
     - 保留多模块但单进程运行：结构仍复杂，边界成本高，否决。
     - 保留 gateway 作为入口：与“无网关”约束冲突，否决。

2. **通信模型决策：全部本地调用**
   - 方案：移除远程调用相关依赖与适配层，改为同模块内 service/repository 调用。
   - 理由：单体应用中远程调用属于无效复杂度，会增加故障点与调试成本。
   - 备选方案：
     - 保留远程调用接口以“兼容未来拆分”：短期收益低，当前复杂度高，否决。

3. **配置体系决策：本地化配置**
   - 方案：以 `application.yaml`（及必要 profile）作为唯一配置来源，移除 Nacos 等外部配置中心耦合。
   - 理由：符合独立部署与本地可运行目标，减少外部基础设施依赖。
   - 备选方案：
     - 保留远端配置中心开关：引入额外条件分支，违背“独立单体”原则，否决。

4. **构建交付决策：单产物交付**
   - 方案：调整父 `pom` 与模块依赖，构建产物收敛为单个可执行 Spring Boot 包；部署脚本和容器配置仅面向该产物。
   - 理由：统一 CI/CD 路径，降低发布失败面。
   - 备选方案：
     - 多产物联合部署：维护成本高，不符合目标，否决。

5. **迁移实施决策：分阶段迁移+阶段验收**
   - 方案：先收敛结构，再迁移代码，再清理依赖与配置，最后做回归与文档更新。
   - 理由：降低一次性大改风险，便于定位问题并快速回滚。

## Risks / Trade-offs

- [风险] 迁移过程中类路径或 Bean 装配变化导致启动失败  
  → 缓解：每阶段执行编译+启动检查，优先修复自动装配与扫描路径。

- [风险] 删除远程调用后，部分功能调用链重写不完整  
  → 缓解：为关键业务链路补充集成测试/冒烟测试，按接口清单逐项核对。

- [风险] 配置中心移除后环境参数缺失  
  → 缓解：建立本地配置基线模板，按环境最小必要参数清单补齐。

- [权衡] 单体收敛会降低未来快速拆分的即刻便利性  
  → 缓解：通过清晰包结构和服务边界命名保留后续演进空间，但不保留分布式运行依赖。

## Migration Plan

1. 在父工程中确认仅保留 `main` 模块为目标结构，标记待迁移模块与待删除模块。
2. 将 `admin`、`project` 的有效业务代码迁移到 `main`，统一包结构与配置命名。
3. 删除 `gateway` 及所有网关相关配置、脚本、文档引用。
4. 移除 Nacos 与远程调用相关依赖、配置项、适配代码。
5. 收敛构建与部署：确保只产出并发布单个 Spring Boot 应用。
6. 执行编译、单测/集成测试与关键路径冒烟，修复问题后清理冗余文件。
7. 更新文档（架构、部署、运维）为单体基线说明。

回滚策略（按阶段）：
- 若某阶段失败，回退该阶段改动并恢复上一步可运行状态，不进行跨阶段叠加回滚。

- 历史脚本与文档中涉及多模块术语是否需要一次性全量替换，还是分批治理？
