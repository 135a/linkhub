## Context

当前仓库将重构为“前后端分离 + 单仓库协作 + 单套部署编排”的单体形态：前端作为独立静态应用，后端作为独立 API 服务，通过 Nginx 统一对外入口。核心约束包括：

- 保持现有业务能力连续可用，避免一次性大改导致不可回滚。
- 本地开发、测试、线上运行环境尽量一致，降低“环境相关问题”。
- 项目定位为面向大厂实习的个人工程作品，文案和呈现需去营销化。
- 需要快速验证改造收益，必须具备基础可测性和可观测性。

相关角色包括：项目维护者（本人）、评审者（面试官/技术评委）、潜在协作者（代码 reviewer）。

## Goals / Non-Goals

**Goals:**
- 明确前后端边界，形成稳定 API 契约与独立构建流程。
- 提供 Docker Compose 一键运行能力，覆盖前端、后端、Nginx。
- 实现 `shortlink.nym.asia` 指向前端并支持 API 反向代理。
- 清理营销化信息，统一项目叙事为工程能力展示。
- 建立最小可用测试与指标体系，支持“有数据地说明质量改进”。

**Non-Goals:**
- 不在本次重构中引入微服务拆分或多仓库治理。
- 不追求一次性覆盖所有自动化测试类型（如完整 e2e 全链路平台化）。
- 不在本次变更中进行大规模业务功能新增。
- 不强依赖复杂云原生组件（如 service mesh、分布式链路追踪平台）。

## Decisions

1. 采用“前端静态站点 + 后端 API + Nginx 网关”的分离单体部署模式  
   - 方案：前端构建产物由 Nginx 托管；`/api` 路径反代后端服务；其余路径走前端路由回退到 `index.html`。  
   - 原因：实现简单、部署成熟、与主流面试场景契合，便于说明工程抽象与边界治理。  
   - 备选：Node 直出前端 + 反向代理。未选原因是前后职责边界不清晰，静态资源缓存策略也更难标准化。

2. 使用 Docker Compose 管理开发/部署一致性  
   - 方案：定义 `frontend`、`backend`、`nginx` 三服务，统一网络、环境变量与健康检查；支持 `docker compose up -d` 启动。  
   - 原因：学习成本低、可复现性高，适合单项目全栈协作。  
   - 备选：仅 Dockerfile 手工启动。未选原因是编排和依赖关系易出错，不利于自动化测试。

3. 以环境变量驱动域名与 API 路径  
   - 方案：前端通过构建时变量注入 API Base URL，Nginx 使用模板化 server 配置支持 `shortlink.nym.asia` 与本地环境切换。  
   - 原因：避免硬编码，提高部署灵活性。  
   - 备选：代码中写死域名。未选原因是不可迁移、难测试。

4. 去营销化策略采用“内容字典 + 全仓扫描”两步治理  
   - 方案：先定义需替换关键词清单（如“拿个Offer”“马丁”），再在前端页面、文档、元信息中统一替换为中性表达。  
   - 原因：可追踪、可复用，后续新增页面也可复检。  
   - 备选：人工零散修改。未选原因是漏改概率高，回归难。

5. 建立“最小可测 + 最小可观测”基线  
   - 方案：  
     - 测试：后端接口单元/集成测试、前端关键页面渲染与交互测试。  
     - 可观测：后端结构化日志、基础业务指标（短链创建成功率、跳转成功率、4xx/5xx 比例、响应延迟分位数）。  
   - 原因：满足“可量化改进”目标，且实现复杂度可控。  
   - 备选：只做日志不做测试，或只做测试不做指标。未选原因是无法形成完整质量闭环。

## Risks / Trade-offs

- [Nginx 路由配置错误导致前端或 API 不可达] → 增加启动前配置校验与容器健康检查，提供本地 smoke test 脚本。  
- [前后端分离后出现 CORS/鉴权问题] → 优先使用同域反向代理避免跨域，鉴权头在网关层透明透传并加测试覆盖。  
- [Docker 化后镜像体积偏大、构建变慢] → 采用多阶段构建与基础镜像优化，分离 dev/prod 配置。  
- [去营销化改动误伤业务文案] → 建立替换白名单/黑名单并由人工 review 关键页面。  
- [指标采集增加运行开销] → 先引入轻量指标，按需扩展高成本观测项。

## Migration Plan

1. 目录重构：明确 `frontend/`、`backend/`、`deploy/nginx/`（或等价目录）职责。  
2. API 契约稳定化：统一后端路由前缀，前端改为基于环境变量访问 API。  
3. 容器化：补充各服务 Dockerfile 与 Compose 编排，加入健康检查。  
4. 网关接入：完成 Nginx `server_name shortlink.nym.asia`、静态托管、`/api` 反代与 SPA 回退规则。  
5. 内容治理：执行营销词替换并完成页面/文档巡检。  
6. 质量基线：补齐关键测试与指标输出，执行一次端到端冒烟。  
7. 灰度上线：先在预发布环境验证，再切换生产；保留旧部署配置用于快速回滚。

回滚策略：
- 保留旧版本镜像与 Compose 配置标签；若异常，回退至上一稳定镜像与网关配置。  
- 域名切换前先验证新栈健康；若失败，Nginx 指向旧上游并恢复旧静态资源版本。
