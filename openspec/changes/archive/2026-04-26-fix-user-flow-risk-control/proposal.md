## Why

目前系统在流量限制（风控控制）模块存在三个严重的问题，影响了开发和最终用户的体验。首先，前端在触发流控时无任何弹窗提示，仅在控制台报错；其次，后端日志中错误记录了被流控拦截的用户名（登录为 admin 却记录成了 other）；最后，系统在单用户开发阶段且 `max-access-count` 设置为 1000 的情况下，仍然频繁触发流控拦截（日志显示请求次数为 null）。这些问题导致风控流控能力不仅起不到保护系统的作用，反而严重阻碍了正常访问，必须予以修复。

## What Changes

- **前端交互变更**：修复前端应用在接收到 A000300（当前访问网站人数过多，请稍后再试）等风控流控错误码时无提示的问题，增加明确的弹窗或 Toast 提示。
- **后端日志与标识变更**：修改 `UserFlowRiskControlFilter` 中获取当前用户名的逻辑，确保风控拦截日志中记录真实的用户名，而不是由于上下文丢失或获取失败而回退打印的 "other"。
- **后端计数逻辑变更**：修复 Redis Lua 脚本或流控计数逻辑，解决其在未达阈值或单用户状态下仍然返回拦截且请求次数为 null 的问题，确保流控计数与拦截判定的正确性。

## Capabilities

### New Capabilities
- `user-flow-risk-control`: 用户请求流量限制与风控管理机制。规定用户维度的流量访问限制、限流计数的判定机制，以及前后端针对流控超限（限流拦截）情况的交互响应规范。

### Modified Capabilities
- 无

## Impact

- **前端系统**：全局请求响应拦截器（用于统一处理错误码 A000300）。
- **后端系统**：
  - `UserFlowRiskControlFilter` 过滤器中的用户名获取、日志打印逻辑。
  - 流控的核心计数代码（可能涉及到 Redis Lua 脚本或对应的 Service 判定逻辑）。
  - `UserFlowRiskControlConfiguration` 及相关属性配置读取机制。
