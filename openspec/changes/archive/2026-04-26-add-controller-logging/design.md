## Context
当前系统在生产环境或日常调试中，由于缺乏细粒度且明确的接口访问日志，难以快速确认外部请求是否成功流转到了后端的 Controller 并执行完毕。为了提高系统的可观测性和问题排查效率，决定在每个 Controller 接口返回结果之前增加明确的日志打印。

## Goals / Non-Goals

**Goals:**
- 在项目中所有对外暴露的 Controller 接口方法的 return 语句之前，统一增加 `log.info` 打印。
- 日志内容需包含当前执行的方法名或请求路径，以明确标识哪一个接口被成功执行。

**Non-Goals:**
- 不修改或破坏任何现有的核心业务逻辑。
- 不引入复杂的分布式链路追踪（Tracing）框架，仅保持轻量级的应用级别日志记录。
- 不涉及 Service 层及以下各层的详细业务日志修改。

## Decisions

- **实现方式选择：直接修改代码 vs AOP (面向切面编程)**
  - **决定**: 采用**直接在代码中修改**的方式，在每个 Controller 方法体内显式写入 `log.info(...)`。
  - **原因**: 这严格遵循了提案中“给每一个接口返回前写一个日志”的需求，直观、清晰且不隐藏逻辑。开发人员在阅读每个接口代码时，可以明确看到日志输出的点。
  - **替代方案**: 考虑过使用 Spring AOP 或 `HandlerInterceptor` 进行统一的无侵入式拦截打印。如果后续发现日志需要带上全局统一的链路 ID 或参数，我们可以再考虑重构为 AOP，但当前阶段保持代码显式直观。

- **日志格式标准**
  - **决定**: 统一定义日志输出格式，例如 `log.info("请求接口 {} 处理完成", "方法名")`。
  - **原因**: 统一的格式有助于后续通过日志收集工具（如 ELK、Loki）进行正则匹配或关键字过滤。

## Risks / Trade-offs

- **[Risk] 手动修改引发的疏漏或代码冲突**
  - **Mitigation**: 由于涉及的 Controller 文件可能较多，实现时需按模块或目录系统性地进行修改，并在提交前进行代码比对（Diff）检查，确保没有误改其他业务逻辑。
- **[Risk] 日志量激增带来的磁盘/IO 压力**
  - **Mitigation**: 仅打印 `info` 级别，且不建议在日志中打印全量且庞大的响应 Body 数据，仅打印接口调用的标识性信息即可。
