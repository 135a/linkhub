## Context

当前系统在短链接分组管理和创建流程中存在以下缺陷：
- **删除逻辑缺陷**：删除分组时，由于前端传递参数或后端校验逻辑不一致，导致触发“分组标识不能为空”的异常。同时，UI 层面未能在操作成功后即时清理本地缓存的状态。
- **编码不一致**：分组名称在数据库存储、后端处理和前端展示的某个环节发生了字符集转换错误，导致中文名称出现乱码。
- **交互反馈缺失**：创建短链接是一个异步过程，但前端在收到成功响应后，未能同步更新 UI（如关闭对话窗、刷新列表），导致用户感知滞后。

## Goals / Non-Goals

**Goals:**
- 实现健壮的分组删除逻辑，解决校验报错并确保 UI 状态一致性。
- 确立全链路 UTF-8 编码规范，彻底消除乱码。
- 优化创建流程的交互体验，确保“即创即见”。

**Non-Goals:**
- 不修改底层的短链接哈希算法。
- 不对整体页面布局进行重新设计。

## Decisions

### 1. 后端：增强校验与编码统一
- **分组删除**: 在 `GroupServiceImpl` 的 `deleteGroup` 方法中，显式校验 `gid` 参数。如果参数为空或无效，返回明确的客户端异常。
- **字符编码**: 
  - 在 `application.yaml` 的数据源 URL 中添加 `characterEncoding=UTF-8` 参数。
  - 配置 Spring MVC 的 `CharacterEncodingFilter`，确保请求和响应均使用 UTF-8 编码。

### 2. 前端：状态同步与回调优化
- **状态管理**: 
  - 在删除分组成功后，通过 Vue 的响应式数组操作（如 `splice`）或调用列表刷新函数，立即更新前端视图。
  - 监听创建短链接接口的 `success` 事件，在弹窗组件中执行 `close` 操作并发送事件通知父组件刷新列表。
- **异常捕获**: 增加全局错误处理拦截，捕获类似“分组标识不能为空”的提示，并在 UI 上以更友好的方式呈现。

## Risks / Trade-offs

- **数据库兼容性**: 如果数据库实例本身未配置为支持 UTF-8，仅修改连接参数可能无法完全解决问题。
- **并发风险**: 在删除分组的同时如果发生其他对该分组的操作，需要确保事务的隔离性。
