# ShortLink 分布式高可用架构说明

## 📊 架构概览

本配置将 ShortLink 从单机部署升级为**分布式高可用架构**,所有关键组件都实现了多实例部署和故障自动转移。

---

## 🏗️ 架构组件

### 1. **MySQL 主从复制**
```
mysql-master (3306)  ← 写操作
    ↓ 二进制日志同步
mysql-slave (3307)   ← 读操作(只读)
```

**特性:**
- ✅ 主从数据实时同步
- ✅ 读写分离(需配置 ShardingSphere)
- ✅ 主节点故障可手动切换

**注意:** 当前为基础主从,生产环境建议使用 MySQL Group Replication 或 MHA

---

### 2. **Redis Sentinel 哨兵模式**
```
redis-master (6379)
    ↓ 复制
redis-slave-1 (6380)
redis-slave-2 (6381)

redis-sentinel-1 (26379)  ← 监控 + 自动故障转移
redis-sentinel-2 (26380)
redis-sentinel-3 (26381)
```

**特性:**
- ✅ 主从数据复制
- ✅ 自动故障检测(< 5秒)
- ✅ 自动选举新主节点
- ✅ 客户端自动重连

**容错能力:** 允许 1 个 Sentinel 节点宕机

---

### 3. **Nacos 集群模式(3节点)**
```
nacos-1 (8848)  ← 服务注册与发现
nacos-2 (8849)     配置中心
nacos-3 (8850)
    ↓ 数据持久化到 MySQL
mysql-master
```

**特性:**
- ✅ CAP 理论中的 AP 模式
- ✅ Raft 协议保证一致性
- ✅ 配置集中管理
- ✅ 服务健康检查

**容错能力:** 允许 1 个节点宕机(多数派原则)

---

### 4. **RocketMQ 双主模式**
```
rocketmq-namesrv-1 (9876)  ← NameServer 集群
rocketmq-namesrv-2 (9877)

rocketmq-broker-master-1 (10911)  ← Broker 主节点1
rocketmq-broker-master-2 (10913)  ← Broker 主节点2
```

**特性:**
- ✅ NameServer 无状态,水平扩展
- ✅ Broker 主节点独立工作
- ✅ 消息分散存储
- ⚠️ 当前未配置从节点(生产环境建议添加)

---

### 5. **应用服务多实例**

#### Gateway (3实例)
```
gateway-1 (8000)  ← Nginx 负载均衡
gateway-2 (8010)     最少连接数算法
gateway-3 (8020)
    ↓ 注册到
Nacos 集群
```

**特性:**
- ✅ Nginx 自动负载均衡
- ✅ 健康检查 + 故障自动剔除
- ✅ Spring Cloud LoadBalancer 客户端负载均衡

#### Admin (2实例)
```
admin-1 (8002)
admin-2 (8012)
    ↓ 注册到
Nacos 集群
```

#### Project (3实例)
```
project-1 (8001)
project-2 (8011)  ← 核心业务服务
project-3 (8021)
    ↓ 注册到
Nacos 集群
```

## 🔧 启动方式

### 1. 首次启动(完整构建)
```bash
docker-compose up -d --build
```

### 2. 查看服务状态
```bash
docker-compose ps
```

### 3. 查看日志
```bash
# 查看所有服务日志
docker-compose logs -f

# 查看特定服务
docker-compose logs -f gateway-1
docker-compose logs -f redis-sentinel-1
```

### 4. 停止服务
```bash
docker-compose down
```

### 5. 重启单个服务
```bash
docker-compose restart gateway-1
```

## 🧪 高可用性测试

### 测试1: Gateway 故障转移
```bash
# 1. 停止一个 Gateway 实例
docker-compose stop gateway-1

# 2. 前端访问不受影响(Nginx 自动切换到其他实例)
curl http://localhost:80/api/short-link/admin/v1/user/login

# 3. 恢复实例
docker-compose start gateway-1
```

### 测试2: Redis 主节点故障
```bash
# 1. 停止 Redis Master
docker-compose stop redis-master

# 2. Sentinel 自动选举新主(约5-10秒)
docker-compose logs -f redis-sentinel-1 | grep "+switch-master"

# 3. 应用自动重连到新主节点(无需重启)
```

### 测试3: Nacos 节点故障
```bash
# 1. 停止一个 Nacos 节点
docker-compose stop nacos-2

# 2. 服务注册发现正常(其他节点继续工作)
# 3. 恢复节点后自动加入集群
docker-compose start nacos-2
```

### 测试4: Project 服务扩容
```bash
# 在 docker-compose.yml 中添加 project-4, project-5...
# 重启后 Nacos 自动发现新实例,负载均衡自动分发流量
docker-compose up -d
```

## ⚙️ 配置说明

### 环境变量覆盖
如果需要修改配置,可以在 `docker-compose.yml` 中设置环境变量:

```yaml
environment:
  - SPRING_CLOUD_NACOS_DISCOVERY_SERVER_ADDR=nacos-1:8848,nacos-2:8848,nacos-3:8848
  - SPRING_DATA_REDIS_SENTINEL_MASTER=mymaster
  - SPRING_DATA_REDIS_SENTINEL_NODES=redis-sentinel-1:26379,redis-sentinel-2:26379,redis-sentinel-3:26379
```

### 资源限制调整
根据服务器配置调整内存限制:

```yaml
deploy:
  resources:
    limits:
      memory: 1g      # 最大内存
    reservations:
      memory: 512m    # 预留内存
```

## ⚠️ 注意事项

### 1. 资源需求
**最低配置:** 8GB RAM + 4 CPU
- MySQL: 1GB × 2 = 2GB
- Redis: 512MB × 5 = 2.5GB
- Nacos: 1GB × 3 = 3GB
- RocketMQ: 1.5GB × 4 = 6GB
- Gateway: 768MB × 3 = 2.3GB
- Admin: 768MB × 2 = 1.5GB
- Project: 1GB × 3 = 3GB
- Frontend: 512MB

**总计:** 约 20GB (可通过减少实例数量降低)

### 2. 网络端口冲突
确保以下端口未被占用:
- 80, 8000-8021, 8848-8850, 9848-9853
- 3306-3307, 6379-6381, 26379-26381
- 9876-9877, 10909-10913

### 3. 数据持久化
所有数据存储在 Docker Volume 中:
```bash
# 查看卷
docker volume ls | grep shortlink

# 备份数据
docker run --rm -v shortlink-mysql-master-data:/data -v $(pwd):/backup alpine tar czf /backup/mysql-backup.tar.gz /data
```

### 4. 生产环境建议
- ✅ 使用外部 MySQL 集群(如阿里云 RDS)
- ✅ 使用外部 Redis 集群(如阿里云 Redis)
- ✅ 使用 Kubernetes 替代 Docker Compose
- ✅ 配置 HTTPS 证书
- ✅ 启用 Nacos 认证
- ✅ 配置监控告警(Prometheus + Grafana)
- ✅ 配置日志收集(ELK)
